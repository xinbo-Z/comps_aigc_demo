## Context

当前 `sci-comp-documention` 首页已经具备基础介绍、组件入口与 Quick Start 内容，但整体仍以平均化信息卡片为主，更接近说明型目录页，而不是成熟组件库文档站的首页入口。与此同时，组件页侧已经逐步建立增强型骨架、结构化案例、API 主表与源码说明能力，首页需要与这套能力形成一致的入口叙事：既帮助用户快速理解项目定位，也帮助用户按合适路径进入基础组件或高阶能力页面。

首页重设计的约束较明确：一方面必须继续保持中文表达、企业级可信感与文档站导航价值，另一方面不能为了视觉展示引入与项目能力不匹配的营销化表达或复杂交互依赖。

## Goals / Non-Goals

**Goals:**

- 将首页升级为“企业级组件体系入口页”，而不是普通目录页
- 重组首页信息层级，明确 Hero、能力背书、浏览路径、核心组件分层、高阶能力亮点与 Quick Start 的职责
- 强化 `SchemaForm` 等差异化能力在首页的可见度，体现基础 wrapper 与高阶能力并存的体系定位
- 保持首页与现有文档体系一致：中文、克制、可维护、便于后续继续演进
- 建立可验证的首页改造成功标准，确保用户能够在首屏或最短浏览路径内理解项目定位、找到主路径入口，并识别高阶能力亮点

**Non-Goals:**

- 不在本次变更中改造全站导航、侧边栏或其他组件详情页
- 不引入在线编辑器、复杂动画系统或额外第三方视觉依赖
- 不将首页改造成营销官网或脱离文档站语境的品牌宣传页
- 不扩大为整站视觉主题重做
- 不在本次变更中调整已有组件详情页的信息结构、案例体系与组件页增强骨架

## Decisions

### 1. 采用“企业级首页 + 轻产品化 Hero”的总体方向

首页整体风格以 Ant Design / Arco 式企业级可信感为主，混入少量 MUI / Chakra 式产品化首屏表达。

- 选择该方向的原因：项目本质是基于 antd v6 的业务组件体系，核心卖点是稳定封装、类型能力、工程一致性与高阶表单能力，而不是大众化 UI 平台或视觉秀场。
- 替代方案：走更强营销感或炫技型首页路线。
- 不选原因：会放大视觉承诺与实际能力规模之间的落差，也会削弱文档站的导航价值。

### 2. 首页按“入口职责”而不是“内容堆叠”组织

首页拆分为六类信息层：Hero、能力背书、推荐浏览路径、核心组件入口、高阶能力亮点、Quick Start。

- 选择该结构的原因：当前首页的主要问题不是内容不足，而是各区块视觉权重过于平均，用户难以快速理解“先看什么、为什么看”。
- 替代方案：保留现有区块，只做样式增强。
- 不选原因：只能改善观感，无法解决首页缺少浏览分流与重点抬升的问题。

### 3. 组件入口采用分层展示，而非全量平铺同权

首页将组件划分为 Featured Components 与 More Components，并单独抬升 `SchemaForm` 作为高阶能力亮点。

- 选择该方式的原因：当前组件数量不大，但能力层次已经开始分化；首页需要用有判断的排序帮助用户建立认知。
- 替代方案：维持所有组件统一卡片网格。
- 不选原因：会继续稀释核心入口与差异化能力的辨识度。
- `SchemaForm` 被单独抬升，不是因为其文档最近完成，而是因为它最能体现该组件库区别于普通 antd wrapper 集合的高阶能力定位；首页需要用它来建立“基础组件 + 高阶配置驱动能力”并存的整体认知。

### 4. Hero 右侧使用静态组件拼贴或展示卡，而不是继续放纯文本面板

首屏右侧使用由现有组件语义拼装而成的静态展示区，例如 Button、Table、Form、SchemaForm、Progress 等预览式卡片。

- 选择该方式的原因：它能在不引入复杂交互成本的前提下，提升首页的产品感与组件感。
- 替代方案：保留当前文字型 panel。
- 不选原因：缺少成熟组件库首页常见的视觉记忆点。
- 展示区至少应同时体现 2~3 类不同语义的组件能力，例如基础交互（Button）、数据展示（Table 或 Progress）与高阶表单（Form 或 SchemaForm），以避免 Hero 退化为单一卡片或纯文字说明区域。

### 5. 实现上优先沿用现有 `HomePage.tsx` 内联样式与 token 体系

本次实现优先复用 `@sci-comp/core` token 与当前页面结构，不额外引入新的样式系统；必要时可将首页内的展示模块拆成局部函数组件，但不扩展为全局设计系统重构。

- 选择该方式的原因：变更聚焦、回归成本低、便于快速验证首页方向。
- 替代方案：同步抽离大规模样式层或新增首页专用公共组件。
- 不选原因：会扩大实现范围，并把首页改造变成新的基础设施重构。

### 6. 首页改造成功标准

首页改造完成后，应至少满足以下判断标准：

- 用户在首屏即可明确识别这是面向业务中台与复杂表单场景的组件体系文档站，而不是普通说明型目录页
- 用户无需依赖滚动浏览完整组件列表，即可找到快速开始、组件浏览入口与高阶能力入口
- 核心组件入口与补充组件入口之间存在清晰层级，而不是所有组件继续同权平铺
- `SchemaForm` 或同等级高阶能力在首页中具有独立且清晰的定位表达
- 首页视觉增强不能削弱其作为文档入口页的导航清晰度与可维护性

## Risks / Trade-offs

- [首页视觉表达变强后，可能削弱“文档入口”的清晰度] → 保持 CTA、路径卡与组件入口的显式导航语义，不做纯展示型海报
- [过度强调少数组件，可能让其他组件显得边缘化] → 保留 More Components 区，并确保所有现有组件仍可从首页进入
- [Hero 拼贴若设计过满，会影响可维护性] → 采用静态 mock 与轻量卡片组合，不接入复杂状态逻辑
- [首页区块变多，移动端容易显得拥挤] → 使用响应式双栏/单栏切换，并控制每个区块的信息密度

## Context

当前组件文档站以 Rspress + MDX 为基础，组件页主要通过 `docs/components/*.mdx` 直接编排说明文字，并借助 [PreviewBlock](../../../apps/sci-comp-documention/doc-components/PreviewBlock.tsx) 与 [demos.tsx](../../../apps/sci-comp-documention/doc-components/demos.tsx) 提供少量实时预览。这个结构已经足够支撑“基础说明 + 单个 demo”的页面，但它还不具备像 Ant Design 官方文档那样组织多案例、承载 API 表格、展示源码说明以及支持代码展开/复制的能力。

现有 `standardize-component-documentation` 已经定义了统一中文章节结构和 Progress 试点方向，但其设计前提仍偏向“轻量属性摘要”，尚未把结构化示例模型、API 表格承载方式和代码展示能力纳入同一套文档架构。新的需求实际上是在现有模板规范之上，再向“组件文档平台能力”迈进一步。

当前讨论还额外明确了第一版边界：增强型组件页必须共享核心必选骨架；示例元数据从第一版起就按未来全站复用标准设计；API 表按业务高频使用口径筛选，而不是只覆盖 wrapper 独有增强字段；案例预览与源码展示必须共享单一真源；每个案例都需要提供接近官网风格的完整源码说明；Button 与 Progress 的试点案例覆盖要达到基础、变体、状态与组合场景等较完整层次；同时需要在元数据或组件接口层为第二阶段在线编辑能力预留扩展位。

约束也比较明确：

- 第一版需要在不引入完整在线代码执行依赖的前提下落地
- 不能复制 Ant Design 官方全量 API，而应突出 wrapper 组件最常用、最关键的属性
- 需要优先试点 Progress 与 Button，两者都属于轻量 wrapper，适合验证统一模型
- 现有 MDX 内容和 demo 代码应尽量复用，而不是推翻重写整个文档站

## Goals / Non-Goals

**Goals:**

- 为增强型组件页定义统一的文档组件骨架，使页面能够承载头部信息、适用场景、封装说明、案例区和 API 表格
- 定义结构化示例元数据模型，让单个案例能够同时关联预览、说明、源码和相关属性信息
- 定义第一版 API 表格策略，覆盖 Progress 与 Button 的常用属性展示方式
- 定义代码展示能力的分级策略，并在第一版中落地源码说明、代码展开和复制能力
- 以 Progress 与 Button 为试点，验证统一文档骨架与多案例组织方式
- 为第二阶段可选的在线编辑能力预留扩展点，而不在第一版强制落地

**Non-Goals:**

- 不在第一版中引入完整的浏览器内在线代码编译与即时执行系统
- 不尝试从 TypeScript 类型全自动生成覆盖所有 props 的完整 API 文档
- 不在本次变更中重写所有组件页，只聚焦文档能力建设与 Progress / Button 试点
- 不修改 `@sci-comp/core` 运行时代码或组件对外 API

## Decisions

### 1. 采用“核心必选骨架 + 可裁剪区块”的增强架构，而不是弱约束模板

**Decision:** 保持现有 Rspress + MDX 页面承载方式不变，在 `doc-components/` 下补充面向增强型组件页的文档原子组件，并将组件定义、适用场景、案例区、API 表与封装说明定义为核心必选区块；其余内容允许按组件类型裁剪。

**Why:**

- 当前用户已经明确要求增强型组件页采用统一核心骨架，而不是仅给出推荐章节顺序。
- 统一核心骨架更有利于后续在不同组件页之间维持稳定阅读习惯与信息对齐。
- 通过保留可裁剪区块，仍能避免简单组件页被过重模板绑死。

**Alternatives considered:**

- 引入新的文档框架或现成 playground 体系：能力更强，但迁移成本和不确定性过高。
- 继续只用现有 `PreviewBlock` 拼接页面：实现快，但无法支撑 API 表和统一案例结构。
- 只提供弱约束模板：初期灵活，但很容易重新滑回页面结构分裂。

### 2. 将“示例组件”升级为“全站复用的示例元数据对象”，而不是试点期临时模型

**Decision:** 第一版文档案例不再只由 React demo 组件构成，而是定义一套按未来全站复用设计的结构化元数据模型，至少包含标题、说明、源码、预览节点、相关属性标记，以及面向在线编辑预留的扩展字段。

**Why:**

- 用户已明确要求示例模型从第一版起就按全站标准设计，而不是只为 Button / Progress 临时抽象。
- 若试点阶段先做临时模型，后续推广到 Input、Tabs、Modal 时很容易出现字段不兼容和二次迁移。
- 为第二阶段在线编辑保留接口位，要求第一版模型至少具备可扩展结构，而不是纯展示型数据。

**Alternatives considered:**

- 在每个 MDX 页面手写所有案例说明与代码块：灵活但重复度高，页面之间难统一。
- 继续集中在 `demos.tsx` 中只导出预览组件：无法支撑结构化案例能力。
- 只为试点定义最小模型：落地更快，但后续全站收敛成本更高。

### 3. 第一版 API 表采用“高频优先”的半结构化维护，而不是只覆盖封装增强字段

**Decision:** 对 Progress 与 Button 试点页，API 表格采用人工维护的半结构化数据源，统一展示属性名、说明、类型、默认值、版本/备注等核心字段；属性筛选按业务高频使用口径进行，而不是仅展示 wrapper 独有增强字段。

**Why:**

- 用户已经明确 API 表应以“业务里高频会用到”为准，即使该属性来自 antd 透传，也可以进入文档主表。
- 当前 wrapper 组件高度依赖 Ant Design 透传，全自动类型提取很容易得到大量不适合直接展示的字段。
- 半结构化数据源既能保证统一展示样式，又能控制维护成本，是当前阶段最稳妥的折中方案。

**Alternatives considered:**

- 手写 Markdown 表格：简单但难复用，后续容易风格分裂。
- 全自动 TS 类型抽取：长期理想，但实现复杂且不适合作为第一版前置条件。
- 只展示 wrapper 增强属性：维护成本低，但无法满足实际查阅需要。

### 4. 代码能力按“完整说明 + 展开/复制 → 在线编辑”分级推进

**Decision:** 第一版增强型组件页强制支持完整源码说明、代码展开与复制，并要求案例预览与源码展示共享单一真源；在线编辑能力作为第二阶段可选能力单独演进，但第一版需在元数据或组件接口层预留扩展位。

**Why:**

- 用户已经明确每个案例的源码说明应接近官网风格，而不是一句简短提示。
- 单一真源可以避免案例运行效果与展示代码漂移，是后续长期维护的必要前提。
- 在线编辑需要浏览器内编译、运行隔离和依赖注入，复杂度明显高于源码展示，因此应延后到第二阶段，但接口位必须先留好。

**Alternatives considered:**

- 第一版直接做完整 playground：体验理想，但会吞掉大量实现成本。
- 只展示静态代码块：满足最低需求，但无法达到“像官网一样可展开查阅”的目标。
- 允许预览与展示源码分别维护：初期编写快，但后期漂移风险很高。

### 5. 试点页面选择 Progress + Button，并按较完整案例层次覆盖

**Decision:** 第一版试点同时覆盖 Progress 与 Button 两个组件页，且案例范围按较完整覆盖建设，至少需要覆盖基础、变体、状态与组合场景等层次，而不是只做 2-3 个最小案例。

**Why:**

- Progress 已经是现有文档中最完整的页面，适合承载多案例和 API 表的增强需求。
- Button 的交互与属性更简单，适合验证 API 表与案例组织是否足够轻量、是否能成为可复制模板。
- 用户已明确希望试点案例层次尽量接近官网常见展示方式，因此需要较完整覆盖而非最低可用演示。

**Alternatives considered:**

- 只试点 Progress：验证深度足够，但无法检验模板在更简单组件上的复用性。
- 直接扩到所有现有组件页：范围过大，难以及时校验设计是否合理。
- 只做最小案例覆盖：实现更快，但不足以验证统一骨架和全站模型是否真的成立。

## Risks / Trade-offs

- [统一核心骨架定义过重，导致简单组件页编写成本上升] → Mitigation：将骨架组件拆成可组合的小块，但仍强制共享核心区块
- [全站标准示例模型在第一版设计过深，拖慢试点落地] → Mitigation：先收敛基础字段与扩展字段边界，避免在第一版加入编辑器运行时细节
- [API 表按高频口径筛选后，不同维护者对“高频”理解不一致] → Mitigation：在任务阶段补充筛选标准，并要求试点页用相同口径整理字段
- [单一真源导致案例组织方式需要调整，增加初期改造成本] → Mitigation：优先围绕现有 demos 代码重组，避免同时改动文档页与组件源码结构
- [第一版不做在线编辑导致与官网体验仍有差距] → Mitigation：通过完整源码说明、展开/复制与多案例组织先补齐高频查阅需求，并在接口层预留第二阶段扩展位
- [Progress 与 Button 的试点模式无法完全迁移到 Form/Table/SchemaForm] → Mitigation：在设计中明确当前试点主要面向轻量 wrapper，复杂组件后续需要变体模板

## Migration Plan

1. 在 OpenSpec 中新增组件文档增强能力 spec，并扩展 `component-doc-standardization` 与 `progress-docs` 的要求
2. 定义增强型组件页的核心必选骨架、全站示例元数据结构、API 数据约定与在线编辑扩展字段边界
3. 先将 Progress 与 Button 页面迁移到增强型结构，补齐较完整案例、多案例组织、API 表、完整源码说明与代码展开能力
4. 基于试点结果，评估是否将相同模式推广到 Input、Modal、Tabs 等轻量 wrapper 组件
5. 若后续需要第二阶段在线编辑能力，再基于已存在的案例元数据与接口预留结构继续演进

本次变更主要影响文档站内容组织与文档组件，不涉及运行时组件数据迁移。
如需回滚，可保留新增元数据与文档原子组件代码但让试点页面退回基础 PreviewBlock 结构，或整体撤销增强型组件页接入。

## Open Questions

- 第一版 API 表中的字段是否需要纳入“版本”或“封装备注”列，还是保持最小四列即可
- Button 与 Progress 的“较完整案例覆盖”在实现阶段如何落成一份可复用的案例分层清单
- 第二阶段在线编辑能力若启动，是否优先做“受控 props 面板”而不是完整自由代码编辑器

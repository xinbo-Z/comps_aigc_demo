## Context

当前组件文档站已经覆盖 Button、Input、Form、Modal、Tabs、Progress 等页面，但整体仍以“单页各自成立”为主，尚未形成统一的中文信息架构。现有页面普遍具备标题、一句话定义、实时预览与基础示例，但在“组件定位、封装边界、适用场景、常用属性、选型建议”这些帮助用户快速理解和决策的层次上并不一致。

Progress 页面已经开始向更完整的中文组件页结构演进，天然适合作为首个规范化试点；而 Modal、Tabs、Input 等页面也存在类似特征：它们都属于基于 Ant Design 的轻量 wrapper，但各自带有少量语义扩展或增强点，适合纳入统一模板约束。相比之下，Form、Table、SchemaForm 这类更偏组合式或模式型组件，虽然也需要文档规范，但其重点不完全在属性摘要，而更多在使用模式说明。

因此，这次变更不是单纯补一个组件页面，而是在不引入新文档系统、不复制 Ant Design 官方 API 的前提下，为组件文档建立一套统一的中文模板与适用规则，并通过 Progress 先行验证。

## Goals / Non-Goals

**Goals:**

- 定义组件文档页统一的中文章节结构，覆盖组件定义、组件定位、适用场景、实时预览、基础用法、关键示例、常用属性、选型建议、封装说明等层次
- 明确不同类型组件适用哪些章节，避免所有页面机械套用同一模板
- 将 Progress 作为首个规范化试点页，验证统一模板在轻量 wrapper 场景中的效果
- 为 Modal、Tabs、Input 等相近组件提供后续迁移方向
- 让 OpenSpec 产物本身也保持中文表达，便于后续讨论与维护

**Non-Goals:**

- 不引入新的文档框架、自动化 API 生成系统或独立内容管理机制
- 不要求所有组件页强制拥有完整 API 表
- 不在本次变更中重写所有组件文档页，只定义规范并落一个试点
- 不修改 `@sci-comp/core` 的运行时代码或组件类型设计

## Decisions

### 1. 采用“统一骨架 + 按组件类型裁剪”的中文模板，而不是强制所有页面完全同构

**Decision:** 为组件文档定义统一骨架，但允许根据组件类型裁剪章节。统一骨架包括：组件定义、组件定位/封装说明、适用场景、实时预览、基础用法、关键示例；“常用属性”和“选型建议”则按组件是否需要决策辅助来选择性启用。

**Why:**

- 当前组件差异明显：Progress、Modal、Tabs 适合强调场景与选型；Form、Table 更适合强调使用模式与结构组织。
- 如果强制所有页面完全一致，简单组件会变重，复杂组件又可能被不合适的模板限制。
- 统一骨架可以建立阅读习惯，选择性章节则保证灵活性。

**Alternatives considered:**

- 完全自由式页面：延续现状，但无法形成统一文档语言。
- 所有页面固定同一章节：一致性高，但不适合组件差异化特点。

### 2. 将 Progress 作为模板试点页，而不是先全量改造全部组件页面

**Decision:** 先在 Progress 页面上落地完整模板所需的核心章节，再根据效果推广到 Modal、Tabs、Input 等页面。

**Why:**

- Progress 组件边界清晰、示例结构完整、常用属性集中，适合承载“中文模板试点”。
- 先试点可以验证页面是否过重、章节是否顺序合理、中文表达是否顺畅，减少一次性大改成本。
- 这与当前探索结论一致：Progress 不应成为特殊页面，而应成为第一张规范化组件页。

**Alternatives considered:**

- 直接批量重写所有组件页：成本高，且在模板未验证前容易返工。
- 仅写规范不做试点：无法检验规范是否可落地。

### 3. 常用属性采用“轻量摘要”，不维护完整 API 表

**Decision:** 在适合的组件页中加入“常用属性”章节，但仅覆盖最常用、最影响选型和理解的属性，不复制 Ant Design 官方的全量参数表。

**Why:**

- 目前多个 wrapper 组件本质上仍高度透传 antd，如果手工维护完整 API，成本高且容易过期。
- 用户在组件库文档中更需要“够用的查阅入口”，而不是再看一遍官方 API 全表。
- 轻量摘要更符合中文文档的阅读路径：先明确用途，再掌握常用入口。

**Alternatives considered:**

- 不写任何属性信息：会降低查阅效率，用户仍需反复跳转。
- 维护完整 API 表：维护成本高，且与当前站点整体粒度不一致。

### 4. 将“封装边界说明”作为规范化组件页的必备信息层

**Decision:** 对于基于 antd 的 wrapper 组件，文档中应明确说明当前组件是轻量透传、语义增强，还是具有明确扩展能力。

**Why:**

- 这是当前文档站最不统一但最关键的信息：用户需要快速判断组件和 antd 原生能力相比增加了什么、限制了什么。
- 对 Progress、Modal、Tabs、Input 这类 wrapper 组件来说，封装边界比完整 API 更能帮助用户理解如何使用。

**Alternatives considered:**

- 仅靠一句话定义隐含边界：不够明确，容易造成误判。
- 把边界信息散落在各示例说明里：不利于快速扫读。

## Risks / Trade-offs

- [模板定义过重，导致简单组件页内容膨胀] → Mitigation：采用“统一骨架 + 可选章节”的方式，仅在需要时补充常用属性和选型建议
- [试点页与其他页面差异突然拉大，短期内出现站点风格不一致] → Mitigation：在设计中明确 Progress 是模板试点，并为下一批迁移页面给出优先级
- [轻量属性摘要被误解为完整 API 覆盖] → Mitigation：在封装说明中明确更多参数仍可按 Ant Design 原生能力传入
- [不同组件类型难以共用一个模板] → Mitigation：按“轻量 wrapper”“语义增强 wrapper”“组合式组件”分层定义适配策略，而不是一刀切

## Migration Plan

1. 在 OpenSpec 中定义组件文档规范化能力与 Progress 文档要求增量
2. 以 Progress 页面为试点，落地统一中文模板中的关键章节
3. 根据试点结果，梳理 Modal、Tabs、Input 的后续迁移策略
4. 在后续实现阶段逐步将规范扩展到更多组件页

本次变更以文档结构和规范为主，不涉及数据迁移。
如需回滚，可撤销试点页面新增章节，并保留基础文档页结构。

## Open Questions

- “组件定位”和“封装说明”是否拆分为两个章节，还是在同一章节内合并表达
- 对 Form、Table、SchemaForm 这类复杂组件，是否需要定义一套偏“使用模式说明”的变体模板
- 后续是否需要在文档首页或指南页增加一份“如何阅读组件文档”的说明，以帮助用户理解统一模板

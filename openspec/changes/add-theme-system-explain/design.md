## Context

当前主题系统实现已经完成了运行时能力闭环：`createThemeTokens()` 负责统一主题输入，`createAntdThemeTokens()` 负责驱动 Ant Design wrapper，`createThemeCssVariables()` 负责驱动 CSS Modules / 自定义样式组件，同时还保留了历史变量兼容层。但这些能力主要体现在源码与少量快速开始示例中，缺少稳定的文档入口与维护规则，导致开发者难以判断应该改哪一层 token、如何在业务中做主题 override、何时可以继续使用旧变量，以及文档内容应当在 README、快速开始和权威指南之间如何分工。

本次变更是典型的跨模块文档治理：它同时涉及 `@sci-comp/core` 包 README、Rspress 指南页、文档站导航与文档工作区 README，且目标不是新增运行时功能，而是为现有主题系统建立统一、长期可维护的说明契约。

## Goals / Non-Goals

**Goals:**

- 为主题系统建立单一权威说明入口，清晰解释单一主题源、双输出通道和 token 分层职责。
- 让业务开发能够快速复制最小接入方式，并理解如何通过同一份 theme override 同时影响 AntD wrapper 与 CSS Module wrapper。
- 为组件库维护者补充兼容变量映射、新增 token 判断规则、禁止事项与验证要求，降低未来继续分裂变量体系的风险。
- 明确 README、快速开始、主题指南页三者的内容边界，避免重复维护与多处失真。

**Non-Goals:**

- 不新增或调整主题系统运行时 API。
- 不在本次变更中扩展 dark mode、品牌多主题切换或可视化 playground。
- 不重写所有组件文档页，只在必要位置补充主题系统入口和维护约束。

## Decisions

### 1. 以独立主题指南页作为权威来源

- 决策：在文档站新增独立的主题系统指南页，作为完整规则源。
- 原因：当前主题说明分散在 README、快速开始和源码接口中，任何一个位置都不足以承载完整心智模型和维护规则；单独的指南页更适合沉淀契约、FAQ 与兼容映射。
- 备选方案：仅扩充 `getting-started.md`。未采用，因为快速开始页应服务“最短接入路径”，继续堆砌完整规则会稀释接入效率。

### 2. 采用“三层内容分工”而不是单页包办

- 决策：`packages/sci-comp-core/README.md` 只提供速览与入口；`getting-started.md` 只保留最小接入示例；完整原理、边界和维护规则集中到主题指南页。
- 原因：开发者在不同阶段关注点不同。第一次接入需要最短示例，维护者需要权威规则，README 则应作为仓库级入口而不是完整手册。
- 备选方案：把所有说明集中到 README。未采用，因为 README 很容易失去聚焦，并且不适合承载较长的映射表和维护规则。

### 3. 主题指南页内容围绕“怎么用 + 怎么维护”组织

- 决策：主题指南页的结构以使用路径和维护判断为主，而不是纯 API 罗列。
- 原因：当前痛点不是“找不到函数名”，而是“不知道什么时候改 seed、什么时候看 semantic、什么时候不能继续用旧变量”。文档必须先建立决策心智，再解释接口。
- 备选方案：按源码导出顺序组织文档。未采用，因为这会让文档贴近实现细节，但不能快速回答业务与维护中的高频问题。

### 4. 将兼容变量映射与禁止事项显式文档化

- 决策：在主题指南页中单列兼容层映射表、迁移规则、推荐做法和禁止事项。
- 原因：历史变量兼容层是当前体系的重要迁移契约，如果不显式说明，团队很容易继续在新代码中消费旧变量，导致迁移窗口无限延长。
- 备选方案：只在源码注释中提示。未采用，因为业务开发与文档维护者并不会稳定阅读实现文件。

### 5. 将验证方法纳入文档契约

- 决策：文档中明确说明主题修改后的最低验证项，包括双通道是否同步生效、兼容变量是否仍可用、相关测试与文档构建是否通过。
- 原因：主题系统是桥接式设计，任何只改一半的调整都可能造成 AntD wrapper 与 CSS Modules 表现不一致；验证要求必须前置为团队共识。
- 备选方案：只在任务或 PR 描述中临时说明。未采用，因为这种知识不会稳定沉淀。

## Risks / Trade-offs

- [文档与实现再次偏离] → 在说明中尽量直接引用当前公开导出与现有行为，并把完整规则收敛到单一指南页，减少多处复制。
- [快速开始页面过度膨胀] → 保持快速开始只承载最小接入示例，把概念性内容转移到主题指南页。
- [README 信息过少导致入口不明显] → 在 README 中保留一句话心智模型、最小示例和明确跳转链接。
- [维护规则过于抽象导致难执行] → 将规则写成兼容映射表、推荐/禁止列表与验证 checklist，而不是宽泛原则。

## Context

当前文档站已经通过 Button 与 Progress 页完成了第一轮增强型组件文档页试点。现有增强模式已经具备统一页面骨架、结构化案例元数据、常用属性 API 表、源码说明与代码展开/复制能力，因此这次扩展的重点不再是重新设计整套文档系统，而是把这一模式稳定推广到 Input、Modal、Tabs、Table 与 Form。

与第一轮试点相比，本轮扩展的难点发生了变化：Input 与 Modal 属于轻量 wrapper，适合直接复用现有模式；Tabs 与 Table 虽然实现层不算最重，但在文档层更容易因行为语义或用户预期过高而失控；Form 则处在基础 Form 与高阶 SchemaForm 的边界位置，如果不先收紧文档边界，后续页面很容易偏离“基础 wrapper 不承载复杂 DSL”的项目约束。

当前还存在一个结构性问题：`apps/sci-comp-documention/doc-components/componentDocData.tsx` 与 `componentDocPreviews.tsx` 目前集中承载 Button / Progress 的文档数据与预览组件，继续把 Input、Modal、Tabs、Table、Form 全部堆入同一文件，会让维护成本、review 成本与后续扩展成本持续上升。因此，本次设计除了页面迁移本身，还需要对文档数据与预览组织方式做一次可控的结构决策。

## Goals / Non-Goals

**Goals:**

- 在不改变增强型组件页核心骨架的前提下，将 Input、Modal、Tabs、Table 与 Form 分批迁移到统一的增强页模式。
- 为每个组件页补齐结构化案例、常用属性 API 表、源码说明与代码展开/复制能力。
- 根据组件类型差异，分别控制轻量 wrapper、行为语义型组件、高预期型组件与边界敏感组件的文档组织策略。
- 明确基础 Form 与 SchemaForm 的职责边界，避免增强型 Form 页变成高阶表单能力入口。
- 在推广过程中解决文档数据源与预览组件的扩展性问题，避免集中式文件继续膨胀。

**Non-Goals:**

- 不重新设计 `ComponentDocPage` 的整体章节结构。
- 不将 SchemaForm 在本次变更中默认升级为增强型页面。
- 不补齐各组件对应的 Ant Design 全量 API 镜像。
- 不引入在线代码编辑、即时运行或浏览器内编译能力。
- 不通过一次性“大迁移”同时完成所有组件页，而是接受分批推进与逐批验证。
- 不将首页、组件导航或站点入口的必要同步扩展为全站信息架构重构。

## Decisions

### Decision: 按组件复杂度分三批推进迁移

本次变更采用三批推进，而不是一次性平推所有页面。

- 第一批：Input + Modal
- 第二批：Tabs + Table
- 第三批：Form

这样安排的原因是三类组件的文档风险不同：Input / Modal 适合先验证模式能否推广；Tabs / Table 适合验证增强页在更高信息密度场景下如何保持范围收敛；Form 则必须在前两批沉淀出稳定写法后，再单独处理其与 SchemaForm 的边界表达。

批次在本次设计中主要承担实施顺序、review 分段与阶段复盘的作用，而不是把一个 change 拆成多个独立归档单元。每一批完成后都应回看案例元数据、API 主表组织以及说明字段职责是否仍适用于后续批次；如发现模型不足，允许在进入下一批前做小范围结构修正，但最终仍以全部批次完成作为本 change 的收口口径。

备选方案是将 5 个页面作为一个大 change 一次迁移。没有采用这一方案，是因为它会把轻量页面迁移、复杂组件范围治理以及 Form / SchemaForm 边界问题混在同一轮 review 与任务分解中，降低可控性。

### Decision: 保持统一骨架，但按组件类型控制案例与说明层次

增强型页面仍沿用统一骨架：组件定义、适用场景、结构化案例区、常用属性 API 表、封装说明，以及按需启用的选型建议。

但不同组件不要求在案例数量与说明深度上完全同构：

- Input / Modal 以高频基础场景和轻量增强点为主。
- Tabs 要重点说明 `lazy` 等行为语义，而不仅是视觉差异。
- Table 要重点控制范围，聚焦基础列表、分页、滚动与虚拟滚动等高频场景。
- Form 要重点说明基础布局、提交与校验流程，并明确高阶能力不在本页范围内。

同时，说明字段职责保持统一：

- `selectionTips` 用于表达选型建议、职责分流与“何时应改用其他能力”。
- `wrapperNotes` 用于表达封装边界、透传规则与未纳入主表的能力说明。
- 案例源码说明仅解释当前案例的用途、实现要点与适用边界，不承担整页级选型说明。

备选方案是为每类组件重新设计骨架。没有采用，是因为这样会削弱当前试点已验证的统一阅读路径，并增加后续页面间风格漂移风险。

### Decision: API 主表继续采用“高频优先”而非全量镜像策略

所有新增页面都继续采用第一轮试点已经验证的策略：API 主表只覆盖最常用、最影响理解和选型的属性字段，而不是追求与 antd 官方文档等量齐观。

- Input 重点突出标签、提示、状态与交互基础字段。
- Modal 重点突出 `fullscreen` 与常用开关/回调字段。
- Tabs 重点突出 `lazy`、`type` 与切换控制字段。
- Table 重点突出 `columns`、`dataSource`、`pagination`、`scroll`、`virtualScroll` 等高频字段。
- Form 重点突出 `layout`、`colon`、`requiredMark`、`initialValues`、`onFinish` 等默认行为字段。

对高信息密度组件，优先级进一步约束如下：先保留最能帮助用户完成页面搭建的基础字段与 wrapper 关键语义；排序、筛选、展开行、复杂缓存、动态编排等非主路径能力默认不进入主案例与主 API 表，除非 wrapper 在这些能力上引入了额外语义并且缺失会直接妨碍理解该组件的主用法。

备选方案是按 wrapper props 或 antd 原生 props 自动生成更全的 API 表。没有采用，是因为这会弱化 wrapper 文档的重点表达，并让 Table / Form 等组件文档迅速膨胀。

### Decision: 在第一批页面迁移前按组件拆分文档数据与预览组织

如果本轮仍继续使用单一 `componentDocData.tsx` 与 `componentDocPreviews.tsx` 承载所有组件页数据，文件规模会在第二批和第三批时明显失控。因此本次设计要求在第一批页面迁移前先完成结构收口，并按组件拆分文档数据与预览：

- `component-doc-data/<component>Doc.tsx`
- `component-doc-previews/<component>Previews.tsx`

统一的 `ComponentDocPage` 骨架仍然保留，不会拆成多套页面容器。案例的单一真源以组件级案例定义为中心：同一案例需要同时提供预览渲染入口、源码字符串、案例标题、说明文本与可选属性标记；页面渲染层从该案例定义消费预览与源码，而不是分别维护两套来源。

备选方案是继续集中维护，等所有页面迁完后再重构。没有采用，是因为那样会把“页面迁移”和“数据结构整理”延迟到更难 review 的阶段。

### Decision: 基础 Form 页必须显式表达与 SchemaForm 的边界

Form 是本轮中最容易偏离项目约束的页面，因此设计上要求基础 Form 页在案例层、API 主表与封装说明三个层面都显式收边：

- 案例只覆盖基础布局、校验、默认值、提交流程等通用表单能力。
- API 主表只覆盖基础 Form 高频字段。
- `wrapperNotes` 与必要的 `selectionTips` 要明确：复杂 schema 驱动、动态编排与 DSL 能力不属于基础 Form 页范围。

基础 Form 的判断口径以“是否仍处于 antd Form wrapper 的直接心智内”为准：静态字段布局、基础校验反馈、默认值、提交流程属于本页范围；依赖 schema/config 生成字段、动态编排字段结构、面向复杂业务 DSL 的能力不属于本页范围。页面可以做分流提示，但不承担 SchemaForm 教程或能力总览职责。

备选方案是直接在 Form 页中顺带兼顾 SchemaForm 入口。没有采用，是因为这会让基础 wrapper 与高阶能力页面重新混写，破坏项目在 `CLAUDE.md` 中已经明确的分层原则。

## Risks / Trade-offs

- [文档数据拆分会扩大第一轮改动范围] → 通过仅拆分数据与预览文件、保留统一骨架组件，控制结构调整的影响面。
- [Tabs 页面可能只展示视觉案例，行为语义表达不足] → 在案例与源码说明中明确解释 `lazy` 的职责和典型使用边界。
- [Table 页面容易扩展成 antd Table 全量教程页] → 在 spec、design 与 tasks 中同时强调“高频优先 API 主表 + 透传能力在封装说明中交代”的策略，并优先剔除非主路径能力。
- [Form 页面容易误吸高阶 SchemaForm 能力] → 将 Form 放到最后一批处理，并在 proposal、design、spec 与 tasks 中同时设定边界约束。
- [分批推进会让部分页面在一段时间内仍保留旧模式] → 接受迁移过程中的短期不一致，但确保每批完成后都保持已迁页面内部结构一致，并在进入下一批前完成字段职责与案例模型复盘。

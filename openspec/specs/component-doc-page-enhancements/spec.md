## Purpose

定义组件文档增强页的结构化能力要求，覆盖统一骨架、结构化示例元数据、第一版 API 表格与源码说明/展开能力。

## Requirements

### Requirement: 组件文档增强页 SHALL 提供结构化文档骨架

系统 SHALL 为组件文档增强页提供统一的结构化文档骨架，用于承载页面头部信息、适用场景、示例区、API 表格与封装边界说明等信息层，并允许按组件类型裁剪非必选区块。

#### Scenario: 试点页面使用统一骨架

- **WHEN** 团队为 Button 或 Progress 构建增强型组件文档页
- **THEN** 页面 SHALL 以统一的文档骨架组织组件定义、适用场景、示例区、API 表与封装说明，而不是继续以零散 MDX 片段拼接

#### Scenario: 核心区块属于必选信息层

- **WHEN** 团队实现任一增强型组件文档页
- **THEN** 页面 SHALL 至少包含组件定义、适用场景、结构化案例区、常用属性 API 表与封装说明等核心区块

#### Scenario: 不同组件类型裁剪非必选区块

- **WHEN** 团队将统一骨架应用到不同类型的组件页面
- **THEN** 文档系统 SHALL 允许组件页根据轻量 wrapper、组合式组件等类型裁剪非必选区块，同时保持共享章节顺序与阅读习惯一致

#### Scenario: 推广到更多组件类型时保持统一骨架

- **WHEN** 团队将增强型组件页从 Button / Progress 推广到 Input、Modal、Tabs、Table 或 Form
- **THEN** 页面 SHALL 继续复用统一核心骨架，而不是按组件重新发明不同章节体系

### Requirement: 组件文档增强页 SHALL 支持结构化示例元数据

系统 SHALL 为组件示例定义结构化元数据模型，使每个案例能够同时关联预览组件、案例标题、文字说明、源码内容与相关属性信息。

#### Scenario: 单个案例关联完整信息

- **WHEN** 团队为 Progress 或 Button 编写一个文档案例
- **THEN** 该案例 SHALL 能以统一结构关联预览内容、案例标题、案例说明、源码字符串与可选的相关属性标记

#### Scenario: 示例元数据按全站复用设计

- **WHEN** 团队为第一版试点设计案例元数据结构
- **THEN** 该结构 SHALL 以未来推广到其他组件页为前提，而不是仅服务于 Button 或 Progress 的临时模型

#### Scenario: 案例预览与源码共享单一真源

- **WHEN** 文档系统同时渲染某个案例的实时预览与展开源码
- **THEN** 两者 SHALL 来源于同一份案例源定义，以避免展示代码与运行效果漂移

#### Scenario: 扩展到更多组件时允许按组件拆分数据源

- **WHEN** 团队将增强型组件页推广到更多组件并持续维护案例数据
- **THEN** 文档系统 SHALL 允许在保持统一元数据结构的前提下，按组件拆分文档数据与预览实现，以避免集中式文件持续膨胀

#### Scenario: 后续页面迁移前先完成结构收口

- **WHEN** 团队准备将增强型组件页继续推广到更多组件
- **THEN** 文档系统 SHALL 在进入新的页面迁移前完成案例元数据与预览组织的结构收口，以降低后续批次返工成本

### Requirement: 组件文档增强页 SHALL 支持第一版 API 表格

系统 SHALL 在第一版增强型组件页中提供 API 属性说明表格，用于展示组件最常用、最影响理解和选型的属性字段，并明确该表格不等同于 Ant Design 官方的全量参数镜像。

#### Scenario: 轻量 wrapper 展示常用属性表

- **WHEN** 用户浏览 Button 或 Progress 等轻量 wrapper 组件页
- **THEN** 页面 SHALL 提供覆盖常用属性名、说明、类型与默认值等字段的 API 表格

#### Scenario: API 表格按高频使用口径筛选

- **WHEN** 团队维护第一版增强型组件页的 API 表格
- **THEN** 文档系统 SHALL 允许将业务中高频使用的透传属性纳入表格，而不是仅展示 wrapper 独有增强字段

#### Scenario: API 表格限定第一版维护范围

- **WHEN** 团队维护第一版增强型组件页的 API 表格
- **THEN** 文档系统 SHALL 允许只维护组件最常用、最关键的属性字段，并在封装说明中明确更多能力仍遵循 Ant Design 原生能力或透传规则

#### Scenario: 高频复杂组件页保持主表范围收敛

- **WHEN** 团队为 Tabs、Table 或 Form 等更高信息密度组件维护增强型页面
- **THEN** API 主表 SHALL 继续以高频字段和 wrapper 关键语义为主，而不是扩展成 Ant Design 官方文档的全量参数镜像

#### Scenario: 复杂组件优先排除非主路径能力

- **WHEN** 团队维护 Table、Tabs 或 Form 等更高信息密度组件页
- **THEN** 排序、筛选、复杂缓存、动态编排或同等级非主路径能力 SHALL 默认不进入主案例与主 API 表，除非 wrapper 在这些能力上引入了额外语义且缺失会直接妨碍理解主用法

### Requirement: 组件文档增强页 SHALL 支持源码说明与代码展开能力

系统 SHALL 为增强型组件页中的每个案例提供源码说明与代码展开/复制能力，以提高查阅效率并为第二阶段在线编辑能力预留扩展空间。

#### Scenario: 用户展开案例源码

- **WHEN** 用户浏览某个组件案例并希望查看实现代码
- **THEN** 页面 SHALL 允许用户展开该案例的源码内容，而不要求用户离开当前页面手动查找文档源码

#### Scenario: 案例提供完整源码说明

- **WHEN** 用户阅读某个案例对应的源码说明
- **THEN** 说明内容 SHALL 至少覆盖该案例的用途、实现要点与适用边界，而不是只提供简短标题式描述

#### Scenario: 第一版为在线编辑预留接口扩展位

- **WHEN** 团队实现第一版增强型组件页
- **THEN** 文档系统 SHALL 在案例元数据或相关展示接口层预留可扩展字段，以支持第二阶段接入在线编辑能力

#### Scenario: 第一版不强制在线编辑执行

- **WHEN** 团队实现第一版增强型组件页
- **THEN** 文档系统 SHALL 将源码展开与复制能力纳入必备范围，但 SHALL NOT 要求所有案例在第一版支持在线编辑和即时执行

### Requirement: 组件文档增强页 SHALL 统一说明字段职责

系统 SHALL 为增强型组件页中的选型建议、封装说明与案例源码说明定义稳定职责，以避免不同页面对相同信息层重复承载或表达漂移。

#### Scenario: selectionTips 表达选型与分流

- **WHEN** 页面启用 `selectionTips`
- **THEN** 该区块 SHALL 主要用于表达选型建议、职责边界与分流提示，而不是重复逐个案例的实现说明

#### Scenario: wrapperNotes 表达封装边界与透传规则

- **WHEN** 页面维护 `wrapperNotes`
- **THEN** 该区块 SHALL 主要用于表达 wrapper 的封装边界、透传规则与未纳入主表能力的说明，而不是替代选型建议

#### Scenario: 案例源码说明不承担整页级说明职责

- **WHEN** 页面为单个案例编写源码说明
- **THEN** 说明 SHALL 聚焦当前案例的用途、实现要点与适用边界，而不是承担整页级选型建议或封装总说明

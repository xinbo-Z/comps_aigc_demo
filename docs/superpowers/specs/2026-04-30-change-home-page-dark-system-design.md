# HomePage 暗黑兼容与重设计

## 背景

当前文档站首页位于 [apps/sci-comp-documention/docs/HomePage.tsx](apps/sci-comp-documention/docs/HomePage.tsx)，已经具备较强的视觉表达意图：包含 glow、渐变、悬浮卡片、组件入口和工作台式展示片段。但它的样式实现仍明显建立在浅色假设和 `createThemeTokens()` 的直接拼接之上，大量区域直接使用 `tokens.colorPrimary + 透明度`、`tokens.colorBgContainer` 与局部硬编码渐变，导致以下问题：

1. Rspress 暗黑模式下容易出现层次发灰、发脏或对比不足。
2. 首页视觉语言与已经完成暗黑兼容的文档展示骨架不一致。
3. `HomePage.tsx` 文件过长，内容结构、装饰样式和交互状态高度耦合，后续维护成本高。
4. 当前首页虽然“有视觉元素”，但还不够像真正的品牌展示型 landing page，冲击力和结构完成度都不够。

本次工作目标不是简单修补暗黑颜色，而是在首页范围内完成“品牌展示型 landing page + 亮暗两态兼容”的重设计收口。

## 目标

- 让首页在亮色和暗色模式下都具备明显的品牌展示感。
- 保留首页作为文档入口页的职责，明确引导用户进入“开始使用”“组件浏览”“主题系统”等关键路径。
- 通过更清晰的层次组织，提升首屏冲击力、入口可发现性与整体完成度。
- 将首页实现从“大型单文件样式堆叠”整理成更清晰的首页专用结构，便于后续维护。

## 非目标

- 不扩展到其他文档页的重设计。
- 不修改 `packages/sci-comp-core` 的主题运行时能力。
- 不把本次变更升级成整站视觉系统改造。
- 不引入复杂动画框架或额外设计系统基础设施。

## 用户已确认的设计方向

- 风格定位：品牌展示型。
- 页面目标：混合型——既要吸睛，也要明确导向使用入口。
- 首屏构图：全屏海报式首屏 + 下方快速入口卡片。
- 范围控制：允许调整首页相关样式组织方式，但不扩散到其他文档页。

## 设计方案

### 1. 信息架构

首页保持四段式结构，但改成更鲜明的品牌展示表达：

1. **海报式首屏**
   - 全屏或接近全屏的主视觉区域。
   - 包含品牌级大标题、简要价值描述、双 CTA、少量关键指标。
   - 背景中包含受控的 glow / gradient / 浮层装饰，形成海报感。
   - 配置少量展示型卡片或组件碎片，避免仅剩纯文字 hero。

2. **快速入口区**
   - 紧接首屏下方，放置高辨识度入口卡片。
   - 明确承接首页主任务：开始使用、组件总览、主题系统、设计原则或类似主路径。
   - 入口卡片既承担信息架构作用，也承担视觉“落点”，避免首页只有氛围没有导航效率。

3. **核心特性 / 组件价值展示区**
   - 重构现有 features 和 components 两块，使其更统一。
   - 特性区强调“为什么选择这套组件库”。
   - 组件入口区强调“可以立即进入哪些核心组件文档”。
   - 两块需要共享统一的表层语义和 hover 语言，而不是各自一套视觉规则。

4. **底部收口区**
   - 使用更完整的 CTA 或品牌收口表达，而不是仅保留简单 footer。
   - 在保证不喧宾夺主的前提下，形成“看完首页后下一步去哪里”的明确引导。

### 2. 视觉层级

首页视觉建立四层结构：

1. **基底层**
   - 使用暗色友好的背景体系作为全页基础。
   - 亮暗模式下共用同一层次结构，只调整亮度、透明度和对比度。

2. **氛围层**
   - 使用受控的 radial gradient、glow、弱光斑、局部模糊光团来制造品牌展示感。
   - 氛围层只负责“气质”，不承担信息可读性。

3. **内容表层**
   - 所有真实可读信息都放在清晰的表层卡片或容器中。
   - 暗色模式下必须保证边界、背景、文字层级稳定，不依赖浅色铺底。

4. **高亮交互层**
   - 主按钮、入口卡片 hover、重点数字、标题局部渐变等属于高亮交互层。
   - 这一层负责“品牌记忆点”和“操作感”。

这种分层方式的关键，是先保证暗黑模式可读，再叠加品牌展示装饰，避免装饰层反过来污染可读层。

### 3. 暗黑兼容策略

首页不再继续采用“浅色 token 直接加透明度”的写法作为主策略，而是改为双层语义：

- **基础语义层**：优先消费 Rspress 暴露的 `--rp-c-*` 变量，用于文本、边框、基础表层、弱背景、阴影语义。
- **品牌装饰层**：在基础语义层之上，局部叠加受控的渐变和 glow，用于 hero 背景、入口卡片强调区、品牌标题或局部图形装饰。

这样做的目的：

- 文本、容器、边框、说明信息天然跟随 Rspress 主题切换。
- 炫丽表达只集中在少数展示点，不再污染整页所有卡片和容器。
- 亮色与暗色共享同一结构，只调整装饰层强弱，而不是写两套相互打架的样式。

### 4. 代码组织策略

不会继续把所有数据、样式、section 组件和交互状态都堆在单个 [HomePage.tsx](apps/sci-comp-documention/docs/HomePage.tsx) 中。

建议调整为首页专用的轻量组织方式，例如：

- `HomePage.tsx`：负责页面装配。
- 首页数据文件：维护 feature、入口卡片、组件入口等静态内容。
- 首页 section / card 组件文件：承载 hero、入口卡片、展示卡片等小型视图单元。
- 首页样式语义文件：集中首页自有的 surface / glow / accent 语义。

约束如下：

- 拆分仅服务于首页，不抽象成全站通用基础设施。
- 如果拆分后发现额外文件没有明显价值，应保持克制，避免过度工程化。
- 新组织方式必须让后续继续调暗色、改文案、增删模块时不再需要在超长文件里来回查找。

### 5. 与现有内容的关系

本次不是推翻重写内容，而是重组表达方式：

- 保留“开始使用”“查看组件”等关键 CTA。
- 保留核心特性和组件入口这两类主要信息。
- 可以减少当前过于分散或视觉重复的片段。
- 如果现有工作台式卡片不再适配海报式首屏，可下移为展示性模块，而不是继续占据首屏主导位置。

## 交付边界

本次改动范围限定在首页及其必要的首页专用组织文件中。

允许：

- 调整 [apps/sci-comp-documention/docs/HomePage.tsx](apps/sci-comp-documention/docs/HomePage.tsx)
- 新增少量首页专用组件、数据或样式语义文件
- 为首页重设计补充必要验证

不允许：

- 修改 `packages/sci-comp-core` 的主题运行时
- 顺手改其他组件文档页的设计
- 扩展成整站统一首页框架

## 验证方案

### 自动验证

至少执行：

- `pnpm --filter sci-comp-documention typecheck`
- `pnpm --filter sci-comp-documention build`

如首页相关测试值得补充，可在文档测试工作区增加最小必要验证，但不强制把这次工作扩展成完整视觉快照体系。

### 人工验收

需要在 Rspress 亮色 / 暗色主题下重点检查：首页的：

- 首屏海报区
- 主 CTA 与次 CTA
- 快速入口卡片
- 核心特性区
- 组件入口区
- 底部收口区
- glow / gradient / hover 是否在暗色下发灰、发脏、失焦或喧宾夺主
- 标题、正文、次要文字、边框、背景层级是否保持清晰

## 风险与控制

### 风险 1：视觉过度而损害可读性

控制方式：先确定内容表层和文字对比，再叠加装饰层。

### 风险 2：首页变成营销页，偏离文档入口职责

控制方式：保留明确 CTA 和快速入口区，确保用户能快速进入实际文档路径。

### 风险 3：实现范围蔓延到全站

控制方式：只允许首页及首页专用组织文件变动，不碰其他文档页。

### 风险 4：单文件继续膨胀

控制方式：按首页专用职责拆分为少量文件，但避免抽象成全局体系。

## 推荐实施结论

采用“首页重组型”方案：

- 保留首页的文档入口职责
- 重做视觉语义和布局组织
- 以“全屏海报式首屏 + 下方快速入口卡片”为核心构图
- 用 `--rp-c-*` 承担基础暗黑兼容，用受控的品牌 glow / gradient 承担展示感
- 将首页整理为少量专用结构，提升后续维护性

这是当前范围内最平衡的做法：既能明显提升观感，也不会把问题扩大成整站重做。
